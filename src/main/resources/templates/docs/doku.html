A01

Broken Access Control · Bibliotheksverwaltung

Zugriffsentscheidungen werden an eine ID aus dem Request festgemacht statt am angemeldeten Benutzer (IDOR). Wer die ID manipuliert, sieht fremde Daten.

Demo

/demo/a01 → „verwundbar": als alice angemeldet owner=bob eingeben → GET /demo/a01?mode=vulnerable&owner=bob liefert Bobs Ausleihen aus.

Gegenmassnahme

Sicherer Pfad ignoriert den Owner-Parameter und liest nur Daten des Principals (findByOwnerUsername(auth.getName())). Privilegierte Sichten per @PreAuthorize("hasRole('ADMIN')") — ROLE_USER erhält 403. Default-Deny.

Code: web/DemoController.java, config/SecurityConfig.java
A05

Injection · Produktkatalog mit Suche

Die Sucheingabe wird direkt in den SQL-String verkettet — so lässt sich die WHERE-Bedingung umschreiben.

Demo

Payload %' OR '1'='1 hebelt den listed = TRUE-Filter aus; interne Produkte (INTERN …) tauchen auf.

... WHERE listed = TRUE
    AND name LIKE '%%' OR '1'='1'

Gegenmassnahme

Parameterbindung über Spring Data JPA — Eingaben werden nie als SQL interpretiert:

@Query("... like lower(concat('%', :term, '%'))")
List<Product> searchListed(@Param("term") String t);
Code: demo/VulnerableProductDao.java, demo/ProductRepository.java
A04

Cryptographic Failures · Passwortspeicherung

Passwörter im Klartext oder mit schnellem, ungesalzenem Hash (MD5) sind bei einem DB-Leak trivial wiederherstellbar; gleiches Passwort ergibt gleichen Hash.

Demo

/demo/a04 → Zeichenkette eingeben und Klartext, MD5 und BCrypt nebeneinander vergleichen. BCrypt ist bei jedem Aufruf anders (Salt).

Gegenmassnahme

BCryptPasswordEncoder als PasswordEncoder-Bean; echte Anmeldedaten nur als BCrypt-Hash gespeichert. HTTPS erzwungen, HSTS gesetzt.

Code: config/SecurityConfig.java, user/DataSeeder.java, demo/CryptoDemoService.java
A02

Security Misconfiguration · Actuator & Header

Unkonfiguriert exponiert Spring Boot Actuator /actuator/env (Secrets) oder /actuator/shutdown (DoS). Fehlende Security-Header öffnen XSS/Clickjacking.

Nachweis

/actuator/health → 200 (UP), /actuator/env & /actuator/beans → gesperrt. Header: curl -I ….

Gegenmassnahme

Actuator-Exposure auf health begrenzt, shutdown deaktiviert. Security-Header zentral im HeadersConfigurer (CSP, HSTS, X-Frame-Options DENY, nosniff …). H2-Konsole aus.

Code: config/SecurityConfig.java, application.yml
A07

Authentication Failures · Brute-Force / Credential Stuffing

Ohne Schutz lassen sich Passwörter automatisiert durchprobieren bzw. geleakte Zugangsdaten breit testen.

Demo

Direkt am Login: nach 5 falschen Versuchen „Konto vorübergehend gesperrt"; viele Versuche von einer IP lösen die Rate-Limit-Meldung aus.

Gegenmassnahme

LoginGuardFilter läuft vor der Passwortprüfung: IP-Rate-Limit, CAPTCHA, Account-Lockout (5 Versuche → 15 Min). Erfolgreicher Login setzt den Zähler zurück.

Code: security/LoginGuardFilter.java, security/LoginAttemptService.java, security/CaptchaService.java
A10

Mishandling of Exceptional Conditions

Stack-Traces im HTTP-Response verraten interne Klassen, Pfade und Versionen — wertvolle Aufklärung für Angreifer.

Demo

/demo/a10 → beide Buttons teilen durch 0. „verwundbar" zeigt den geleakten Stack-Trace, „sicher" löst dieselbe Exception generisch behandelt aus.

Gegenmassnahme

Globales @ControllerAdvice gibt dem Client nur eine generische Meldung + Korrelations-ID; voller Stack-Trace nur ins Server-Log. include-stacktrace=never.

Code: web/GlobalExceptionHandler.java, application.yml
A09

Security Logging (Querschnitt)

Ohne Logging von Login-Versuchen fehlen Audit-Trails; werden Passwörter/Tokens geloggt, entsteht ein neues Leck.

Gegenmassnahme

Dedizierter SECURITY_AUDIT-Logger (eigener Appender, JSON-ähnlich) für LOGIN_SUCCESS/FAILURE, ACCOUNT_LOCKED u. a. — niemals Passwort/CAPTCHA/Token.

Details

Eigene Seite mit Live-Beispiel: Logging-Doku öffnen →

Code: security/AuthEventListener.java, logback-spring.xml
