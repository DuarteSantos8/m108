Sicheres Logging

SLF4J + Logback mit dediziertem Sicherheits-Audit-Strom — sicherheitsrelevante Ereignisse strukturiert protokolliert, ohne je ein Geheimnis preiszugeben.

Bewertungskriterium: Logging korrekt und sicher implementiert
Appender Audit-Ereignisse Log-Level Keine Secrets Live-Beispiel

Zwei Appender (Logback)

Konfiguriert in logback-spring.xml. Der Audit-Strom ist bewusst vom Anwendungs-Log getrennt (additivity="false"), damit er separat nach Logstash/Loki abgegriffen werden kann.

STDOUT

Allgemeines Anwendungs-Log im Klartextmuster, von Docker eingesammelt. Root-Logger auf INFO.

AUDIT — SECURITY_AUDIT

Dedizierter, strukturierter Strom (JSON-ähnliche Zeile) nur für sicherheitsrelevante Ereignisse. Quelle: security/AuthEventListener.java.

Protokollierte Sicherheitsereignisse

Logins, Zugriffe und Fehler werden über den SECURITY_AUDIT-Logger erfasst:

EventLevelBedeutung
LOGIN_SUCCESSINFOErfolgreiche Anmeldung (nur Benutzername).
LOGIN_FAILUREWARNFalsche Anmeldedaten.
LOGIN_CAPTCHA_FAILWARNCAPTCHA falsch/abgelaufen.
LOGIN_RATE_LIMITWARNZu viele Versuche von einer IP.
ACCOUNT_LOCKEDWARNKonto nach 5 Fehlversuchen gesperrt (15 Min).

Log-Level sinnvoll eingesetzt

DEBUG INFO WARN ERROR
  • INFO für normalen Betrieb (z. B. erfolgreicher Login, Demo-Daten geladen).
  • WARN für verdächtige/abgewehrte Ereignisse (Fehlversuche, Lockout, abgelehnter Zugriff).
  • ERROR für unerwartete Server-Fehler — vollständiger Stack-Trace nur ins Server-Log (siehe A10).

Keine sensiblen Daten im Log

Niemals geloggt

Passwörter, CAPTCHA-Lösungen, Session-Tokens und personenbezogene Daten (PII) werden bewusst nie protokolliert. Der AuthEventListener schreibt ausschliesslich Benutzername, Ereignistyp und (bei IP-Limits) die Client-IP.
